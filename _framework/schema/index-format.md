# Index Format

How `kb/index.md` is structured and how agents use it together with the two-tier role preload.

## What `kb/index.md` is

A per-kb-directory catalog of the pages within. **Auto-generated by lint** (Rule 15); never hand-edited. Regenerated on every `/check` or `/wrap-up` invocation.

The index is a quick-scan view — summary, type, status, and `relevant_to` for every page. Bodies are not embedded; agents follow wikilinks to load bodies when needed.

## Format

```markdown
# <area-path> kb index

_Auto-generated; do not edit by hand._
_Last regenerated: 2026-05-08_

## Findings (3)

- [[f-2026-05-shot-noise]] · active · high confidence
  Shot noise floor at 1310nm photodetector; dominates above ~10kHz.
  _relevant_to:_ photodetector design, noise budget, 1310nm wavelength

- [[f-2026-04-photodiode-responsivity]] · active · high confidence
  Responsivity calibration for the 1310nm InGaAs detector.
  _relevant_to:_ detector characterization, calibration

- ...

## Decisions (2)

- [[d-2026-05-04-bias-current]] · active
  Bias at 1mA, not 5mA. Trades noise floor for thermal load.
  _relevant_to:_ detector design, thermal budget

- ...

## Concepts under test (5)

- [[c-2026-04-1f-noise-bias]] · under_test · medium confidence
  Whether 1/f noise floor depends on bias direction.
  _relevant_to:_ noise floor characterization

- ...

## Concepts (other) (3)

_(seed, developing, supported, falsified, dropped, superseded — grouped together with status visible per entry)_

- [[c-2026-03-quantum-efficiency]] · supported · high
  ...

## Sources (12)

_(externally-sourced material summaries)_

- [[s-2026-04-saleh-teich-ch17]]
  Saleh & Teich, Photonics, ch 17 — chapter on photodetector noise mechanics.

- ...
```

## Section ordering

By type, in this order:

1. **Findings** (active first, then archived)
2. **Decisions** (active first, then superseded)
3. **Concepts under test** (highlighted because they're work-in-progress)
4. **Concepts (other)** — seed, developing, supported, falsified, dropped, superseded
5. **Sources**

Within each section, pages are ordered by `updated` descending (most recently touched first).

## Per-entry format

```
- [[<id>]] · <status> · <confidence if applicable>
  <summary>
  _relevant_to:_ <comma-separated tags>
```

The wikilink resolves to the page. The status and confidence appear inline so they're visible at scan time. `relevant_to` is included so agents can match against current task topics without opening the page.

For `source` pages, confidence isn't included (sources don't carry confidence); the entry is just id, summary, and a link to the raw file:

```
- [[s-2026-04-saleh-teich-ch17]]
  Saleh & Teich, Photonics, ch 17 — chapter on photodetector noise mechanics.
  _raw:_ [saleh-teich-ch17.pdf](../raw/papers/saleh-teich-ch17.pdf)
```

## How agents use the index

Agents reach for `kb/index.md` in three situations:

1. **Discovery within an area.** "What concepts are we tracking?" → load `<area>/kb/index.md` and scan.
2. **Cross-area orientation.** When the role allows but discourages deep cross-area reads, the other area's `kb/index.md` is the appropriate level of access.
3. **Finding superseded pages.** When updating a page, the agent can scan the index for pages marked `superseded` whose `superseded_by` points at the current page — useful for updating in-bound links.

## Relationship to the role's preload list

A role's preload has two tiers:

- **Full preload** — small, curated; bodies loaded. Schema docs, brief, POR, pulse, and a handful of frequently-referenced kb pages.
- **Frontmatter preload** — broad; only frontmatter blocks loaded. Specified as directory patterns (e.g., `/areas/research/optics/kb/`).

The agent's initial context is "full preload bodies" plus "frontmatter blocks for files matching the frontmatter preload patterns." `kb/index.md` is referenced from the role's full preload (it's small, structured, and useful at session start), but the index is also the canonical navigation aid agents reach for during work — agents can read another area's index without it being in their preload.

## When the index goes stale

The linter regenerates the index on every run. If you make edits and don't run lint, the index will be out of sync with reality. The `/wrap-up` skill always runs lint, so end-of-session it's correct. Mid-session staleness is fine — agents who need it fresh can run `/check`.

## Per-directory indices for sub-areas

Each kb directory gets its own index. For `areas/research/`, that's:

- `areas/research/kb/index.md` (parent area's general kb)
- `areas/research/optics/kb/index.md` (sub-area optics)
- `areas/research/electronics/kb/index.md` (sub-area electronics, if it exists)

The parent area's index does **not** automatically include sub-area pages — sub-areas have their own indices. A role working in `optics` typically loads both `areas/research/kb/index.md` and `areas/research/optics/kb/index.md` (explicit inheritance via preload list).
